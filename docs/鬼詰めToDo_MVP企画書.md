# 引き継ぎ：鬼詰めToDo MVP企画書（実装フェーズ向け）

元の企画メモ（失敗時UXに全振りしたTODO案）を前提に読むこと。本ファイルは「思い出し魔人」「もうひとりのジブン」の引き継ぎmdと同じ作法（app-planningスキルの7章構成）で詰めた叩き台。

> **★スタンス確定**：このアプリは **ネタ／発表が一次、プロダクト化は二次以下**。よって判断に迷ったら **「実用か」より「面白いか」を優先する**（実用の物差しに引っ張られて笑いを殺さない＝一戸さんの「面白さファースト」の原則どおり）。3章の入力動機問題などは"正論の弱点"として記録はするが、ネタ前提では無視してよい。

---

## 0. 一行サマリ

**期限を守らせない。期限を破った後にこそ動き出す**TODOアプリ。普通のTODOが「期限前の通知・整理」を支援するのに対し、鬼詰めは **「期限切れ後の体験」に全振り** する。期限を過ぎると、妙に丁寧な「進捗確認担当」から電話がかかってくる。MVPは **「期限切れ→着信1イベント」だけ** に絞る。

---

## 1. 解決する問題

```
既存解の前提：「通知で思い出させれば、人は期限前に動く」
前提が崩れる瞬間：通知を見ても動かない。未来の自分への約束は軽い。
このアプリが狙う場所：期限を破った"後"。失敗を失敗のまま終わらせず、イベントの開始条件にする。
```

着想は「人は通知では動かないが、**他人からの期待・社会的圧力・見られている感覚** では動く」という非対称性。そこで **「期限を過ぎたら誰かが困る世界」を人工的に作る**。一般的なTODOでは期限切れ＝失敗状態だが、鬼詰めでは期限切れ＝イベント発火条件になる。

---

## 2. 使用シーン

- **いつ（入力）**：「これは絶対やらなきゃ」という1件を、自分にプレッシャーをかけたくて登録する瞬間。低頻度・気まぐれでよい。
- **いつ（本編＝着信）**：登録した期限を過ぎた、**アプリを開いていない**任意のタイミング。仕事中・夕方・週末など。「忘れてた頃に来る」が効く。
- **どこ**：スマホがある場所ならどこでも。本編はむしろ**アプリの外**で起きるのが理想。
- **心理状態**：着信が鳴った瞬間の「うわ、来た」＋「いや電話てｗ」の同居。怖さと笑いが半々。
- **発表デモ時**：登壇者がその場で「期限＝1分後」のタスクを仕込み、トークの合間に**通知が来る→タップで着信画面が立ち上がる**（MVP＝②の手触り）。①に昇格できれば"自分から鳴る"になり、掴みはさらに強い。

---

## 3. 起動動機（★スキルが「最も抜けやすい」と名指しした箇所・最重要）

このアプリは普通のTODOと **起動動機が逆転している**。

- 普通のTODO：管理・確認のために **ユーザーが開く**。
- 鬼詰め：**アプリのほうから来る**（着信・留守電）。**再接触のエンジンは着信そのもの。** ユーザーの意思に依存せず再起動がかかる ＝ これがコンセプトの強み。

ただし「最初の一回＝タスクを入れる動機（入力動機）」は別問題（思い出し魔人で指摘された「起動動機の罠」と同じ構造）。期限を破りがちな人ほどタスクを入れない、という矛盾を構造的に抱える。**ただしネタ一次の今は弱点扱いしない**——プロダクト化を本気で目指す段階3で初めて効いてくる論点として、記録だけしておく。

```
トリガー（入力）：「これは詰められたい」1件を自分で登録。デモでは登壇者が期限直近で仕込む。
入口：TechConf発表 →「なんだこれｗ」→ 触る。アイコン／口コミ。
再接触トリガー（本編）：期限切れ → 着信が自動で来る（意思に依存しない）。← 普通のTODOと違う芯
開く頻度：入力は低頻度でよい。本編の頻度は"期限"が駆動する。
```

**MVP（発表デモ前提）での割り切り**：入力動機の弱さは**今は解かない**。デモは「仕込んだタスクが期限切れ→着信」を見せれば成立する。プロダクト化フェーズで初めて「どうやってタスクを入れさせ続けるか」が本丸の課題になる、と正直に残しておく。

---

## 4. 体験のゴールとアプリの思想

- **ゴール**：着信が鳴った瞬間に、ユーザーが「うわｗ なんだこのアプリ」と声に出すこと。怖さと可笑しさが同時に来る一瞬を作れたら勝ち。
- **思想（このアプリの人格）**：**「失敗時UXに全振り」「期限切れ後が本編」**。
  - 派生する判断の羅針盤 → **「怒る」ではなく「確認する」**。怖い上司ではなく、妙に丁寧な進捗確認担当。
  - 「思い出し魔人」「もうひとりのジブン」と同じ血筋＝**ネタの皮で真面目を運ぶ**。電話というガワで「社会的圧力で人は動く」という真面目を運ぶ。
- **トーンの軸（毒抜き）**：NG＝暴言・人格否定・恫喝・強すぎるパワハラ表現。OK＝妙に丁寧・事務的な圧・認識合わせ・心配している風。**「丁寧なのに逆に怖い」が笑いの源泉**で、ここがリアルな恫喝に振れると凶器化する（もうひとりのジブンの「リアル＝凶器／デフォルメ＝お守り」と同じ）。

## 5. MVP構成

### 必要なもの（最小で体験が成立する要素）

- **タスク作成**：入力は **タスク名 ＋ 期限日時のみ**。
- **タスク一覧**：状態（`未期限` / `期限切れ`）を表示。
- **期限切れ検知**：期限超過をローカルで判定し、着信イベントの対象にする。
- **着信イベント**：期限切れ後に着信を発生。**本物の電話発信はしない**（着信画面"風"UI）。
- **着信画面**：発信者名（＝進捗確認担当）／着信中アニメーション／`出る`・`無視` の2ボタン。
- **応答**：
  - `出る` → 定型セリフ再生 → 終了。**音声（ElevenLabsで事前生成）を主**、テキストは字幕として併記（＝音声OFF環境・聞き取り保険のフォールバック）。
  - `無視` → **留守電が残る**（留守電の一覧 or バッジで「残ってる感」を出す）。留守電も同じく事前生成音声＋テキスト。
- **キャラクター**：1人だけ。**「進捗確認担当」**（肩書きだけ存在する、実在しそうで実在しない人物）。
- **ローカル保存**。

#### セリフ叩き台（固定文・1キャラ）

- 着信→出た：「お疲れ様です。〇〇の件、進捗確認のお電話です。現状未完了という認識で相違ありませんでしょうか。」
- 無視→留守電：「お世話になっております。〇〇について確認したくご連絡いたしました。折り返しお待ちしております。」

### 意図的に外したもの（なぜ外したかも一言）

- **AI会話**（理由：MVPの仮説は「事務的な定型文でも圧が出るか」。むしろ固定文のほうが世界観が固まる。会話は当たってから）
- **音声認識**（理由：応答は`出る`/`無視`の2択で足りる。声を聞き取る必要がない）
- **本物の電話発信**（理由：演出が目的。実発信はリスクとコストだけ増えて笑いは増えない）
- **複数キャラクター**（理由：「進捗確認担当」1人で世界観が立つか先に見る。増やすのは語彙が確定してから）
- **レベル／好感度／ポイント**（理由：ゲーミフィケーションは別の動機を持ち込み、「期限切れ後が本編」の一点集中をぼかす）
- **クラウド同期／共有／カレンダー連携**（理由：ローカルで体験は成立。連携はラクさの改善で仮説と無関係）
- **着信／留守電の履歴、メッセージ、既読スルー演出、放置日数による変化**（理由：いずれも「圧の厚み」を足す方向。1着信が刺さってから乗せる）

## 6. 検証したい仮説（★MVPの存在理由）

```
主仮説（発表用）：「期限切れ後が本編＝失敗時UXに全振り」という発想自体が、見た人に刺さるか（面白い／記憶に残るか）。
副仮説（体験）：暴言ゼロ・事務的な進捗確認の電話"だけ"で、「うわ来た」のプレッシャーと笑いを両立できるか（"丁寧なのに怖い"が成立するか）。
検証方法：自分で数日、本当の自分のタスクで使う ＋ 発表で観客の反応（笑い・ざわつき）を見る。
当たった場合：複数キャラ・圧の段階変化・留守電履歴へ（将来案）。
外れた場合：圧の出し方（丁寧さの度合い・頻度・定型文の質）を調整。それでもダメなら「電話という形式」自体を疑う。
```

## 8. 実装メモ（技術・要検証）

- **技術構成**：Android／Flutter（Dart）。状態管理は最小（`ChangeNotifier` + `ListenableBuilder`）、保存は `shared_preferences`。他2アプリと同じ構成で揃える。
- **着信画面**：Flutterの全画面UIで自作（実在のCallKit的機構は不要）。着信アニメ＋`出る`/`無視`。
- **★期限切れ→着信の"発火"方法（決定済み）**：
  - **MVP＝②（確定）**：期限超過でローカル通知 → タップ → 着信画面を開く（`flutter_local_notifications` 等）。確実・軽い。まずこれで一周作る。
  - **①は格上げ候補（②が動いてから検討）**：アプリ外でも勝手に着信UIが立ち上がる（フルスクリーンインテント）。映える（ステージの"money shot"）が、Android 12+ の `USE_FULL_SCREEN_INTENT`・正確なアラーム（`SCHEDULE_EXACT_ALARM`）・Doze の制約で端末差・挙動差が出やすい。**②で手応えがあり、かつ①が現実的そうなら昇格**。
- **音声（MVP採用・確定）**：固定セリフが数本なので、**ElevenLabsで事前生成した音声ファイルを同梱**（実行時API不要）。"丁寧な声"が笑い／怖さの主成分なので主役扱い。テキストは字幕で併記。セリフを足すたびに生成→差し替えの運用にしておく。
- **デモでの期限切れの起こし方**：実時間で待つ／その場で「期限＝1分後」に設定／デバッグで即発火、のどれか。本番は安定する形を選ぶ（要検証）。
- **データ駆動の余地**：セリフ・キャラ名・着信までの間（ま）は、コード直書きせず差し替えやすく持つ（魔人mdの「assetsにJSON外出し」と同じ思想）。
